# Phase 1: My Profile RAG chatbot — Scaffolding Plan

A personal RAG chatbot that answers employer/recruiter questions using my CV and project data as the knowledge base. Everything runs locally on my Ubuntu machine via Docker.

---

## Open Questions

> [!IMPORTANT]
> **LLM Provider**: ~~Ollama~~ → **Gemini API (free tier)**. The backend will call `gemini-3.5-flash` via the `google-generativeai` Python SDK. A `GEMINI_API_KEY` env var must be set (via `.env` file, git-ignored). Ollama and all `extra_hosts` config are removed.

> [!IMPORTANT]
> **Embedding Model**: Gemini `gemini-3.5-flash` via the same `google-generativeai` SDK — zero extra dependencies, no local model download.

> [!NOTE]
> **Frontend Framework**: Vite + React (TypeScript) scaffolded manually. Chat UI components (**ChatWindow**, **ChatInput**, **MessageBubble**) generated via **[v0.dev by Vercel](https://v0.dev)** and pasted into the project. TailwindCSS **v3** is locked in (v0 always outputs v3 + shadcn/ui).

## Service Breakdown

### 1. `frontend` — Vite + React

| Detail | Value |
|---|---|
| Base image | `node:20-alpine` |
| Dev port | `5173` |
| Build tool | Vite |
| Language | TypeScript |
| Component library | **shadcn/ui** (installed via `npx shadcn@latest init`) |
| Styling | **TailwindCSS v3** ✅ locked in (v0 always outputs v3) |
| UI Generator | **v0.dev by Vercel** — Chat components generated via AI prompt |
| API proxy | Vite dev proxy → `backend:8000` (no CORS issues in dev) |
| Build strategy | Single-stage for dev; multi-stage prod build outputs to `nginx:alpine` |

**Key components:**
- `useChat` hook — hand-written; manages message state
- `api.ts` — hand-written; single source of truth for backend URL
- `ChatWindow`, `ChatInput`, `MessageBubble` — **generated by v0.dev** (see workflow below)
- No Redux / Zustand in Phase 1 — local state only

### v0.dev Workflow

**Step 1 — Scaffold the project first** (Vite + shadcn/ui + Tailwind v3 installed).

**Step 2 — Open [v0.dev](https://v0.dev) and use these prompts:**

#### Prompt A — Full Chat Layout
```
Create a full-page RAG chatbot UI using shadcn/ui and Tailwind CSS.
It should have:
- A dark, professional theme (slate-900 background)
- A fixed sidebar on the left showing the title "Karlis | AI Profile" and a tagline
- A main chat area with a scrollable message history
- User messages right-aligned in a blue bubble
- Bot messages left-aligned with a subtle card style and a small avatar/icon
- A sticky input bar at the bottom with a text field and a Send button
- A "Powered by Gemini" badge in the footer
Export as React TypeScript components.
```

#### Prompt B — Typing Indicator (optional refinement)
```
Add an animated typing indicator (3 bouncing dots) that appears
while the bot is generating a response. Use Tailwind CSS animations.
```

**Step 3 — Copy generated code** into:
- `frontend/src/components/ChatWindow.tsx`
- `frontend/src/components/ChatInput.tsx`
- `frontend/src/components/MessageBubble.tsx`

**Step 4 — Wire up** `useChat.ts` hook and `api.ts` to the generated components.

### 2. `backend` — FastAPI

| Detail | Value |
|---|---|
| Base image | `python:3.11-slim` |
| Dev port | `8000` |
| ASGI server | Uvicorn with `--reload` in dev |
| Validation | Pydantic v2 |
| Vector store client | `chromadb` (HTTP client mode → talks to `chromadb` service) |
| LLM| **Gemini `gemini-3.5-flash`** via `google-generativeai` SDK |

**Key endpoints:**

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/health` | Liveness probe |
| `POST` | `/chat` | Accepts `{ question: str }`, returns `{ answer: str, sources: list }` |
| `POST` | `/ingest` | *(Phase 2)* Triggers re-embedding of `/data` folder |

**RAG flow (per `/chat` request):**
```
1. Embed the incoming question   (Gemini 3 Flash)
2. Query ChromaDB top-k chunks   (k=5 default)
3. Build prompt with context     (Jinja2 template)
4. Call Gemini  (google-generativeai SDK)
5. Return answer + source docs
```

---

### 3. `chromadb` — Vector Store

| Detail | Value |
|---|---|
| Image | `chromadb/chroma:latest` |
| Port | `8001` (internal) |
| Persistence | `./chroma_db` bind-mounted to `/chroma/chroma` |
| Mode | HTTP server (backend connects as client — decoupled) |

## docker-compose.yml (Structure Preview)

```yaml
# 3 services: frontend, backend, chromadb
# 1 custom bridge network: rag-net
# 2 host bind-mounts: ./data (read-only), ./chroma_db (read-write)
# 1 .env file: GEMINI_API_KEY (git-ignored)

services:
  frontend:
    build: ./frontend
    ports: ["5173:5173"]
    depends_on: [backend]
    networks: [rag-net]

  backend:
    build: ./backend
    ports: ["8000:8000"]
    volumes:
      - ./data:/app/data:ro          # CV & project docs, read-only
    environment:
      - CHROMA_HOST=chromadb
      - CHROMA_PORT=8001
      - GEMINI_API_KEY=${GEMINI_API_KEY}  # loaded from .env
    depends_on: [chromadb]
    networks: [rag-net]
    # No extra_hosts needed — Gemini is called over HTTPS (cloud API)

  chromadb:
    image: chromadb/chroma:latest
    ports: ["8001:8001"]
    volumes:
      - ./chroma_db:/chroma/chroma   # Persistent vector store
    networks: [rag-net]

networks:
  rag-net:
    driver: bridge
```

## Verification Plan

### Automated
- `docker compose up --build` — all 3 services reach `healthy` state
- `curl localhost:8000/health` → `{"status": "ok"}`
- `curl -X POST localhost:8000/chat -d '{"question":"Who are you?"}' -H 'Content-Type: application/json'` → returns JSON answer

### Manual
- Open `localhost:5173` in browser → chat UI loads
- Type a question → answer appears in chat window
- Check `./chroma_db/` directory exists and is non-empty after first ingest

### Potential Friction Points *(per modular-coder skill)*
1. **`GEMINI_API_KEY` must be set** in a `.env` file at project root before `docker compose up` — the backend will crash on startup without it
2. **Gemini free tier rate limits**: `gemini` free tier is fine for personal use, but rapid testing may hit limits
3. **`./data/cv.md` must exist** before ingestion — the backend will start but `/chat` will return empty context
4. **v0.dev requires a free Vercel account** — you'll need to be logged in at `v0.dev` to generate and export components
5. **v0 output may need minor wiring** — generated components will use placeholder props/state; `useChat.ts` will adapt them to the real API
